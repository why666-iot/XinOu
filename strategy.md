# AI 使用策略

## 一句话原则

AI 是工具，不是老师。你先想，AI 来纠错。反过来就是在自欺欺人。

---

## 四遍学习法

### 第一遍：跑通

允许大量 AI，不要求理解，系统能跑就行。

### 第二遍：逐行读代码，补语法基础

这一遍的目标不是写注释，而是**读懂每一行在语法层面做了什么**。

具体做法：逐文件阅读，遇到看不懂的语法（指针、回调、结构体、模板、宏等），
停下来让 AI 用最小的例子解释这个语法点，然后在项目代码中找到它的实际用法。

```
你的问题：这一行 `void (*callback)(int event, void *data)` 是什么意思？
AI 解释：这是函数指针，callback 是一个变量，指向一个接受 int 和 void* 参数的函数……
你的动作：在项目里找到 callback 被赋值和被调用的地方，串起来理解
```

每个文件读完后，写一段话总结：这个文件做了什么，对外提供了哪些接口，为什么需要它。

**判断这一遍是否完成的标准：** 随便指一行代码，你能说清楚它的语法含义和在系统中的作用。

### 第三遍：写"为什么"注释 + 破坏性实验

在第二遍已经读懂语法的基础上，这一遍关注设计决策。

注释要回答"为什么这样做"，不是"这行代码做了什么"：
```cpp
// 没用：将音频数据写入缓冲区
// 有用：环形缓冲区解耦 I2S 中断和 WebSocket 发送，防止 DMA 被阻塞
ring_buffer_write(buf, data, len);
```

同时做破坏性实验，主动改参数搞坏系统，再修复：

- 把 `COMMAND_TIMEOUT_MS` 改成 100ms → 理解超时机制
- 环形缓冲区大小减半 → 理解 32KB 的来源
- 关掉 NSNet 降噪 → 理解信号处理的作用
- 断开 WiFi → 理解重连和事件驱动机制

每次实验写一句结论：**"X 的存在是为了解决 Y，去掉它会导致 Z。"**

### 第四遍：独立改造一个功能

不是从零重写整个模块，而是**在现有代码上独立添加或修改一个功能**。

例如：加一个新的命令词、改一下状态机逻辑、加一个 LED 闪烁模式。

规则：自己先写，写不出来的地方再问 AI，但必须理解 AI 给的每一行。

这一遍完成后，你可以说"这个功能是我自己加的"，这在复试中比"我跑通了别人的项目"有说服力得多。

---

## 答辩准备

不要让 AI 出题给你背。正确做法：

1. 你自己列 10 个你觉得最难回答的问题
2. 用自己的话写答案，不查资料
3. 把答案给 AI，让它扮演答辩老师追问你
4. 追问中暴露的漏洞，就是你要补的地方

复试时如果你能说"我试过把缓冲区改成 16KB，结果出现了卡顿，因为……"，
比任何背出来的答案都有说服力。
