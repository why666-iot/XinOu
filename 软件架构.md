# 心偶 AI情感陪伴机器人 — 软件架构设计文档

> 版本：v2.0 | 更新：2026-03-25
> 架构方向：ASR → LLM → TTS 流水线（替代 voice-to-voice）

---

## 1. 项目概述

心偶是一款面向中国市场的 AI 情感陪伴机器人（大创项目），采用端云协同架构：

- **嵌入式端（ESP32-S3）**：负责本地唤醒词检测、录音、BLE配网、音频播放
- **云端（Python 后端）**：负责 ASR 语音识别 → LLM 生成回复 → TTS 语音合成

单个玩偶绑定一个用户，云端通过设备 ID 区分多用户，支持多玩偶并发连接。

### 为什么放弃 voice-to-voice？

| 问题 | voice-to-voice (旧) | ASR+LLM+TTS (新) |
|------|---------------------|------------------|
| 长期记忆 | 无法实现（黑盒，拿不到文字） | 完整文本，可存库检索 |
| 场景路由 | 无法按话题切换提示词 | 先分类再生成，精准路由 |
| 回复质量 | 受限于 Omni 模型 | 可换用最强 LLM |
| 可维护性 | 各环节耦合 | 各模块独立替换 |
| 延迟 | ~2-3s | 流式处理后 ~1-2s |

## 3. 端云通信协议

沿用现有 WebSocket 协议，**ESP32 端无需大改**。

### 连接建立

```
ESP32 → Server: WebSocket 连接
  URL: ws://服务器IP:8888?device_id=AA:BB:CC:DD:EE:FF
```

`device_id` 使用 ESP32 的 BLE MAC 地址（6字节，唯一），服务端据此识别用户。

### 消息类型

| 方向 | 类型 | 内容 | 说明 |
|------|------|------|------|
| ESP32 → Server | Binary | 16kHz 16bit mono PCM | 录音数据，实时流式发送 |
| ESP32 → Server | Text (JSON) | `{"type":"vad_end"}` | VAD检测到说话结束 |
| Server → ESP32 | Binary | 16kHz 16bit mono PCM | TTS生成的语音，流式返回 |
| Server → ESP32 | Ping | — | 音频流结束信号（现有协议保留） |

> **ESP32 端改动极小**：仅需在 WS_URI 中追加 `?device_id=...`，其余代码不变。

## 5. ASR 模块设计

### 接口：阿里云 Paraformer-realtime（流式）

**为什么选 Paraformer**：
- 中文识别率业界领先
- 支持实时流式输入（WebSocket 协议）
- 价格低（~0.8元/小时通话时长）

### 工作流程

```python
# modules/asr.py 核心逻辑（伪代码）

class ASRModule:
    async def transcribe_stream(self, audio_chunks: AsyncIterator[bytes]) -> str:
        """
        接收 PCM 音频块流，返回完整识别文本

        1. 建立到阿里云 ASR 的 WebSocket 连接
        2. 边接收 ESP32 音频块，边转发给 ASR
        3. 收到 vad_end 信号后，等待 ASR 返回最终文本
        4. 关闭 ASR 连接，返回完整文本
        """
```

### 音频格式

ESP32 发送的音频直接转发，无需转码：
- 采样率：16kHz
- 位深：16bit
- 声道：单声道（mono）
- 编码：PCM（裸数据）

## 6. 两次 LLM 调用设计

这是本架构的核心。通过两次 LLM 调用实现"先分类，再精准回复"。

### 第 1 次调用：场景分类

**目标**：判断用户当前话题属于哪个场景，用于选择对应提示词。

```
输入：
  system: [classifier.txt 分类提示词]
  user:   "今天被老板骂了，感觉很委屈..."

输出（强制 JSON）：
  {"scene": "work/workplace_pua", "confidence": 0.92}
```

**设计要点**：
- 使用小快模型（如 Qwen-Turbo、DeepSeek-V3 的轻量版）
- 只输出 JSON，不生成回复文本，速度极快（~0.3s）
- `scene` 字段对应 `prompts/` 目录下的文件路径

**场景分类体系**：

```
daily          - 日常闲聊（默认）
work/
  promotion_failure  - 升职加薪失败
  layoff             - 裁员失业
  workplace_pua      - 职场PUA/被骂
  overtime           - 加班内卷/疲惫
  job_search         - 求职焦虑
family/
  separation_fear    - 离别/分离恐惧
  future_loneliness  - 对未来的迷茫
  family_of_origin   - 原生家庭问题
```

### 第 2 次调用：场景化回复生成

**目标**：基于完整上下文（场景提示词 + 用户记忆 + 对话历史）生成高质量回复。

```
system_prompt 拼接顺序：
  ① base.txt              —— 基础人格设定（~200 tokens，固定）
  ② prompts/{scene}.txt   —— 场景专属话术（~300 tokens，按场景换）
  ③ 用户记忆段落           —— L1画像+L2近期记忆（~500 tokens，动态注入）

messages:
  [最近 10 轮对话历史]
  用户最新输入
```

**流式输出处理**：

```python
# core/orchestrator.py 核心流程（伪代码）

async def handle_turn(device_id, user_text):
    # 第1次调用：分类（快速）
    scene = await llm.classify(user_text)

    # 加载提示词 + 记忆
    system_prompt = build_system_prompt(scene, device_id)

    # 第2次调用：流式回复
    sentence_buffer = ""
    async for token in llm.stream_reply(system_prompt, user_text):
        sentence_buffer += token
        # 检测句子边界（。！？...）
        if is_sentence_end(sentence_buffer):
            # 立即送 TTS，不等待全部生成完
            await tts.synthesize_and_stream(sentence_buffer)
            sentence_buffer = ""

    # 后台异步：提取记忆（不阻塞响应）
    asyncio.create_task(memory.extract_and_save(device_id, user_text, full_reply))
```

## 7. TTS 模块设计

### 接口选型

| 方案 | 优点 | 缺点 | 推荐 |
|------|------|------|------|
| 火山引擎 TTS | 音质自然、情感丰富、中文优秀 | 需注册账号 | ✅ 首选 |
| 阿里云 CosyVoice | 开源可自部署、零样本克隆 | 自部署有硬件成本 | 备选 |
| 讯飞 TTS | 老牌稳定 | 音质略逊 | 不推荐 |

### 流式合成策略

**关键优化**：不等 LLM 生成完整回复，而是**逐句送 TTS**：

```
LLM 输出："今天真的很辛苦，" ──▶ TTS 合成 ──▶ WebSocket 发给 ESP32
LLM 输出："被这样对待确实很难受。" ──▶ TTS 合成 ──▶ WebSocket 发给 ESP32
LLM 输出："你的感受完全可以理解..." ──▶ TTS 合成 ──▶ WebSocket 发给 ESP32
```

**句子分割规则**：遇到 `。！？…` 或长度超过 50 字时切分（避免等太久）。

### 音频格式

TTS 输出需与 ESP32 AudioManager 兼容：
- 采样率：16kHz（或 24kHz，AudioManager 支持）
- 位深：16bit
- 声道：单声道
- 编码：PCM

所有音频块通过 WebSocket Binary 帧发送，结束时发送 Ping 帧（现有协议）。

## 8. 记忆系统设计

### 设计原则

**不使用向量数据库**（pgvector 太重，学生项目不需要）：
- 三层记忆结构 + LLM 压缩
- 纯 SQLite，零运维成本
- 注入量恒定（~500 tokens），不随时间膨胀

### 8.1 SQLite 表结构

```sql
-- database/schema.sql

-- 用户表（设备ID → 用户画像）
CREATE TABLE users (
    user_id         INTEGER PRIMARY KEY AUTOINCREMENT,
    device_id       TEXT UNIQUE NOT NULL,   -- ESP32 BLE MAC地址
    portrait_text   TEXT DEFAULT '',        -- L1 用户画像（自然语言段落）
    portrait_updated_at DATETIME,
    created_at      DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 近期记忆表（L2）
CREATE TABLE memories (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER NOT NULL REFERENCES users(user_id),
    category    TEXT,           -- 记忆类别：event/preference/emotion
    summary     TEXT NOT NULL,  -- 一句话摘要
    importance  INTEGER DEFAULT 5,  -- 重要性 1-10
    is_archived INTEGER DEFAULT 0,  -- 0=近期记忆(L2), 1=已归档(L3)
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);

-- 对话历史表
CREATE TABLE chat_history (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    user_id     INTEGER NOT NULL REFERENCES users(user_id),
    session_id  TEXT NOT NULL,   -- 本次会话UUID（唤醒一次=一个session）
    role        TEXT NOT NULL,   -- 'user' 或 'assistant'
    content     TEXT NOT NULL,
    created_at  DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

### 8.2 三层记忆架构

```
L1 用户画像（users.portrait_text）
  - 自然语言段落，~300 tokens
  - 描述：用户基本信息、性格特征、长期状态、近期主要事件
  - 更新频率：每 10 轮对话压缩一次
  - 示例：
    "用户是一名29岁的产品经理，在北京一家互联网公司工作。
     性格偏内向，工作压力较大，最近几个月在准备跳槽。
     喜欢看书和打游戏，和女朋友异地恋快两年了。
     上周跟直属领导发生了冲突，情绪低落中。"

L2 近期记忆（memories 表，is_archived=0）
  - 最近的重要事件摘要，每条一句话
  - 保留最近 20-30 条
  - 每次注入选取最相关的 3-5 条（~200 tokens）
  - 示例：
    - "用户昨天提到被老板当众批评，很委屈"
    - "用户女朋友下周来北京，用户很期待"
    - "用户最近在看《人生海海》，觉得很有感触"

L3 归档记忆（memories 表，is_archived=1）
  - 已被 L1 吸收的旧记忆
  - 保留但不注入 prompt
  - 未来可做检索增强（Phase 4 再考虑）
```

### 8.3 记忆压缩机制

**触发条件**：每完成 10 轮对话（session_id 下 user 消息数 % 10 == 0）

**压缩流程**：

```
触发压缩（异步后台，不影响当前对话）：
  │
  ├── 1. 取当前 L2 近期记忆（全部 is_archived=0 的条目）
  │
  ├── 2. 取当前 L1 用户画像（portrait_text）
  │
  ├── 3. 调用 LLM 生成新画像：
  │      prompt: "根据以下记忆片段和旧画像，更新用户画像..."
  │      输出: 新的自然语言画像段落（~300 tokens）
  │
  ├── 4. 更新 users.portrait_text
  │
  └── 5. 将已合并的 L2 记忆标记为 is_archived=1
```

### 8.4 记忆注入策略

每次对话时，从 SQLite 取出并注入 system_prompt：

```python
def build_memory_section(user_id) -> str:
    portrait = get_portrait(user_id)          # L1，~300 tokens
    recent = get_recent_memories(user_id, n=5) # L2，最相关5条，~200 tokens

    return f"""
[关于这个用户你需要知道的]
{portrait}

[近期值得记住的事情]
{chr(10).join(f'- {m}' for m in recent)}
"""
```

**注入量恒定**：L1(~300) + L2(~200) = ~500 tokens，无论对话多久都不增长。

## 9. 话题分类 + 场景提示词

### 分类器提示词（prompts/classifier.txt）

```
你是一个话题分类助手。
根据用户的最新一句话，判断属于哪个场景，只输出JSON，不要解释。

可选场景：
- daily（日常闲聊）
- work/promotion_failure（升职加薪失败）
- work/layoff（裁员失业）
- work/workplace_pua（职场PUA/被骂/被批评）
- work/overtime（加班内卷/疲惫）
- work/job_search（求职焦虑）
- family/separation_fear（离别/分离恐惧）
- family/future_loneliness（对未来的迷茫）
- family/family_of_origin（原生家庭问题）

输出格式：{"scene": "场景ID", "confidence": 0.0-1.0}
```

### 场景提示词设计原则

每个场景提示词（如 `prompts/work/workplace_pua.txt`）包含：

1. **共情话术**：如何第一句话让用户感到被理解
2. **避免雷区**：该场景下绝对不说的话（如"你要坚强"）
3. **引导策略**：如何自然地引导用户倾诉更多
4. **结尾处理**：如何收尾，让用户感到温暖

## 10. ESP32 端改动说明

### 必须改动（1处）

在 `main/main.cc` 连接 WebSocket 时附加设备 ID：

```cpp
// 修改前
#define WS_URI_DEFAULT "ws://10.225.67.53:8888"

// 修改后（追加 device_id 参数）
// device_id 在运行时从 esp_read_mac() 获取，拼接到 URI
// websocket_client.cc 中需要支持连接时发送设备ID
```

### 不需要改动

| 模块 | 理由 |
|------|------|
| 音频采集（INMP441 I2S） | 格式不变（16kHz 16bit PCM） |
| 唤醒词检测（WakeNet9） | 本地运行，与云端无关 |
| VAD 检测 | 本地运行，发送 vad_end 信号 |
| 本地命令词（MultiNet7） | 开灯/关灯/拜拜继续本地处理 |
| BLE 配网 | 不影响 |
| AudioManager 流式播放 | 64KB 环形缓冲区已完美支持 |
| WebSocket 二进制收发 | 协议格式不变 |

## 13. 模型选型建议

### ASR（语音识别）

| 方案 | 推荐度 | 理由 |
|------|--------|------|
| 阿里云 Paraformer-realtime | ⭐⭐⭐⭐⭐ | 中文最强，流式 WebSocket，~0.8元/时 |
| 讯飞实时语音转写 | ⭐⭐⭐ | 稳定，但价格略高 |
| 腾讯云实时语音识别 | ⭐⭐⭐ | 可用，中文略逊 |

**选 Paraformer**，文档：https://www.alibabacloud.com/help/zh/model-studio/real-time-speech-recognition

### LLM（语言模型）

| 用途 | 推荐模型 | 理由 |
|------|----------|------|
| 场景分类（第1次调用） | qwen-turbo / deepseek-v3（轻量） | 快速、便宜，只需输出JSON |
| 主对话（第2次调用） | DeepSeek-V3 | 中文超强，性价比极高，~¥0.002/千tokens |
| 记忆提取（异步） | qwen-turbo | 结构化提取，不需要最强模型 |
| 记忆压缩（异步） | 与主对话相同 | 需要理解力 |

**推荐 DeepSeek-V3 做主对话**：中文理解和情感表达出色，价格是 GPT-4 的 1/50。

### TTS（语音合成）

| 方案 | 推荐度 | 理由 |
|------|--------|------|
| 火山引擎 TTS | ⭐⭐⭐⭐⭐ | 音质最自然，情感语音，中文最优，~0.2元/千字 |
| 阿里云 CosyVoice | ⭐⭐⭐⭐ | 开源可自部署，零样本克隆声音 |
| 讯飞语音合成 | ⭐⭐⭐ | 老牌稳定，自然度略逊 |

**选火山引擎**，注册抖音/ByteDance 开发者即可使用，文档完善。

### WebSocket 事件处理（server 端）

```python
@app.websocket("/ws")
async def websocket_endpoint(ws: WebSocket, device_id: str):
    await ws.accept()
    session = SessionManager.create(device_id)

    async for message in ws.iter_bytes():
        if message_is_audio(message):
            session.asr_buffer.append(message)
        elif message_is_vad_end(message):
            # 触发完整处理流程
            await orchestrator.process(session, ws)
```

### 完整对话延迟分解（目标 <2s）

用户说完话（VAD触发）

  ├── 提示词拼接 + 记忆查询：~0.05s

  └── ESP32 开始播放 ≈ 用户说完后 1.5s
